### 2.2 关键接口定义（需提前输出给 B、C）

```python
@dataclass
class StageResult:
    success: bool
    artifacts: dict[str, str] = {}
    metadata: dict[str, Any] = {}
    error: str | None = None

@dataclass
class PipelineEvent:
    event: str       # progress / completed / error
    task_id: str
    video_id: str
    stage: str
    status: str      # running / completed / failed
    progress: int    # 0-100
    message: str
```

### 2.3 建议开发顺序

1. 项目脚手架 + config.yaml + 空 FastAPI 项目跑起来
2. SQLAlchemy ORM 模型 + Alembic 初始化
3. 全局错误处理 + 日志
4. EventBus 事件总线
5. PipelineOrchestrator 骨架（等 B、C 的模块注入）
6. Docker 构建（最后）

## 3. 角色 B：视频处理流水线

### 3.1 职责清单

| 模块 | 具体任务 | 输出产物 | 依赖 |
|------|---------|---------|------|
| VideoProcessor.parse | B 站链接解析（BV/AV/短链接），提取元数据 | app/processor/parser.py | yt-dlp |
| VideoProcessor.download | 画质选择、下载进度 hook、限流控制 | app/processor/downloader.py | yt-dlp |
| VideoProcessor.extract | ffmpeg 提取音频（16kHz, mono, PCM） | app/processor/audio.py | ffmpeg |
| Transcriber 抽象 | 策略模式基类 + 统一返回格式 | app/transcriber/base.py | - |
| FasterWhisperTranscriber | 本地 Whisper 转写 + 进度回调 + GPU 管理 | app/transcriber/whisper.py | faster-whisper |
| BjianTranscriber | 必剪 API 转写实现 | app/transcriber/bjian.py | httpx |
| AutoTranscriber | 自动切换（本地优先，API 兜底） | app/transcriber/auto.py | 以上两个 |
| 重试逻辑 | 指数退避重试 + 降级方案 | app/processor/retry.py | - |
| 文件管理 | 目录结构创建、产物清理、磁盘空间检查 | app/processor/storage.py | - |

### 3.2 模块接口

```python
class VideoProcessor:
    async def parse(self, url: str, video_dir: str) -> StageResult: ...
    async def download(self, video_dir: str, quality: str, progress_cb: Callable) -> StageResult: ...
    async def extract_audio(self, video_dir: str, progress_cb: Callable) -> StageResult: ...

class Transcriber(ABC):
    async def transcribe(self, audio_path: str, video_dir: str, progress_cb: Callable) -> StageResult: ...
```

### 3.3 建议开发顺序

1. VideoProcessor.parse（解析逻辑，最独立）
2. VideoProcessor.download（yt-dlp 封装）
3. VideoProcessor.extract（ffmpeg 封装）
4. FasterWhisperTranscriber（核心转写）
5. BjianTranscriber（备选）
6. AutoTranscriber + 重试逻辑
7. 进度回调接入 EventBus

## 4. 角色 C：AI + 知识库

### 4.1 职责清单

| 模块 | 具体任务 | 输出产物 | 依赖 |
|------|---------|---------|------|
| LLMClient | 统一封装通义千问/DeepSeek 的 chat 和 stream_chat | app/llm/client.py | httpx |
| NoteGenerator | Prompt 构建 + LLM 调用 + Markdown 解析和后处理 | app/note/generator.py | LLMClient |
| VectorStore.chunk | 语义切片（按标题分割 + 合并 + overlap） | app/store/chunker.py | - |
| VectorStore.embed | text-embedding-v3 批量向量化 | app/store/embedder.py | httpx |
| VectorStore.storage | ChromaDB 增删查 | app/store/vector.py | chromadb |
| HybridRetriever | 向量检索 + 关键词检索 + RRF 排序 | app/retriever/hybrid.py | VectorStore |
| QAEngine | 上下文构建 + LLM 问答 + 引用来源标记 | app/qa/engine.py | Retriever + LLMClient |

### 4.2 模块接口

```python
class LLMClient:
    async def chat(self, messages: list[dict], temperature: float = 0.1) -> str: ...
    async def stream_chat(self, messages: list[dict], temperature: float = 0.3) -> AsyncGenerator[str, None]: ...

class NoteGenerator:
    async def generate(self, video_dir: str, video_meta: VideoMeta, progress_cb: Callable) -> StageResult: ...

class VectorStore:
    async def store(self, note_id: int, video_id: str, video_dir: str, progress_cb: Callable) -> StageResult: ...
    async def delete_vectors(self, video_id: str) -> None: ...
    async def similarity_search(self, query: str, top_k: int, filter: dict | None = None) -> list[dict]: ...

class QAEngine:
    async def ask(self, query: str, mode: str, video_id: str | None = None) -> AsyncGenerator[str, None]: ...
```

### 4.3 建议开发顺序

1. LLMClient 封装（最独立，可拿来就测）
2. NoteGenerator（需要 prompt 反复调试）
3. VectorStore 的 chunk + ChromaDB 存储
4. HybridRetriever + RRF 排序
5. QAEngine + SSE 流式输出

## 6. 阶段规划

### 6.1 Phase 1：基础设施（1 天）

| 角色 | 任务 | 产出 |
|------|------|------|
| A | 脚手架、数据模型、配置管理、日志 | FastAPI 空项目能启动 |
| B | 调研 yt-dlp API、ffmpeg 命令行封装 | 下载 + 提音频的独立 demo |
| C | LLMClient 封装、通义千问 API 调通 | 能调通 LLM 的 demo |
| D | Vue 框架搭建、路由、布局、公共组件 | 前端壳子能跑起来 |

**里程碑：** 后端 health check 通过，前端显示页面框架。

### 6.2 Phase 2：核心功能开发（4 天）

| 角色 | 任务 | 产出 |
|------|------|------|
| A | PipelineOrchestrator 骨架 | 流水线能串起来（各阶段先 log）|
| B | VideoProcessor + Transcriber 完成 | 从链接到转写文本的全流程 |
| C | NoteGenerator + VectorStore 完成 | 从转写到笔记到向量的全流程 |
| D | 所有页面开发（Mock 数据） | 完整的前端交互 |

**里程碑：** 可以跑通链接 -> 笔记的端到端流程。

### 6.3 Phase 3：集成联调（1 天）

| 角色 | 任务 | 产出 |
|------|------|------|
| A | 集成 B、C 模块、Docker | 完整后端 |
| B | 配合联调，修 bug | 稳定下载转写 |
| C | HybridRetriever + QAEngine 完成 | 问答功能可用 |
| D | 去掉 Mock 对接真实 API | 前端功能完整 |
| 所有人 | 测试 + bugfix | 可演示的完整系统 |

**里程碑：** 完整端到端流程跑通：贴链接 -> 下载 -> 转写 -> 笔记 -> 问答。

### 6.4 Phase 4：打磨（1天）

- 性能优化（Whisper 模型量化、下载缓存）
- 错误处理完善
- 前端体验打磨（加载态、错误态、动画）
- Docker 镜像构建验证

## 7. 协作约定

### 7.1 接口契约协作

1. 角色 A 先定好接口签名（Python 类型注解或 Protocol class）
2. 角色 B 和 C 各自实现后 export 自己的模块
3. 角色 A 在 PipelineOrchestrator 中组装
4. 角色 D 以 API 接口文档为准，不做端到端等待
